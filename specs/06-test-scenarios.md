# Spec 06 — 테스트 시나리오

## 목적

이 문서는 현재 구현 기준의 수동 테스트 체크리스트다.
`plan.prompt.md`의 Todo를 실제로 검증할 때 기준점으로 사용한다.

## 1. Import 기본 흐름

### 시나리오 1-1. 정상 파일 import
- 입력: 유효한 차샘 JSON 파일
- 기대 결과:
  - 파싱 성공
  - 요약 바에 총 건수/OK/경고/오류 표시
  - 모든 행이 기본 선택 상태로 표시

### 시나리오 1-2. 사용자 미확인 상태
- 입력: 파일은 올렸지만 `UserIdInput` 확인 미완료
- 기대 결과:
  - "다음 단계: 검토하기" 버튼 비활성화
  - 사용자 확인 필요 안내 문구 표시

### 시나리오 1-3. 일부 행만 선택
- 입력: import 후 일부 체크박스 해제
- 기대 결과:
  - 선택 건수가 즉시 갱신됨
  - ReviewPage에는 선택된 행만 전달됨

## 2. 변환/경고 품질

### 시나리오 2-1. `birthdayType = 'N'`
- 입력: `birthdayType: 'N'` 레코드 포함 파일
- 기대 결과:
  - warning 상태로 표시
  - 양력으로 처리되었음을 알리는 경고 메시지 노출

### 시나리오 2-2. 음력 변환 실패
- 입력: 음력 날짜가 유효하지 않아 `lunartosolar()`가 실패하는 레코드
- 기대 결과:
  - error 상태로 표시
  - 저장 대상에서 제외됨

### 시나리오 2-3. 일주 불일치
- 입력: 원본 `ilju`와 계산값이 다른 레코드
- 기대 결과:
  - warning 상태로 표시
  - ReviewPage에서 원본/계산값 차이가 보임

## 3. 병합 그룹

### 시나리오 3-1. 이름 변형 병합
- 입력: 같은 사람으로 보이는 이름 변형(`++`, `정`, 숫자 suffix 등) 2건 이상
- 기대 결과:
  - ReviewPage의 "그룹 병합" 탭에 표시
  - 대표 레코드 1건만 실제 저장 목록에 남음

### 시나리오 3-2. 메모 병합 확인
- 입력:
  - 대표 레코드 `notes = "본원"`
  - 흡수 레코드 `notes = "허자 가능성"`
- 기대 결과:
  - 병합된 대표 레코드 `notes`에 두 내용이 함께 남음
  - 현재 기준으로 `' / '` 연결 형태 확인

### 시나리오 3-3. 윤달 인접 예외
- 입력: 날짜 교집합은 낮지만 음력 변환 시 윤달 인접으로 판단되는 유사 레코드
- 기대 결과:
  - 이름 기준으로 병합 그룹에 포함될 수 있음
  - 병합이 과도하지 않은지 수동 확인 필요

## 4. Review 저장

### 시나리오 4-1. JSON 내보내기
- 입력: ReviewPage에서 저장 방식 `json`
- 기대 결과:
  - split 포맷 JSON 다운로드
  - 각 항목이 `person_basic`, `person_detail` 구조를 가짐

### 시나리오 4-2. DB 저장 성공
- 입력: 정상 레코드만 포함된 목록
- 기대 결과:
  - 성공 건수가 표시됨
  - 실패 건수는 0
  - split mismatch가 없으면 일치 메시지 표시

### 시나리오 4-3. DB 저장 mismatch
- 입력: 서버 저장 결과와 클라이언트 split 결과가 다르도록 만드는 케이스
- 기대 결과:
  - 저장은 성공으로 집계될 수 있음
  - 완료 화면에 mismatch 이름 목록이 별도 경고로 표시됨

## 5. Manual Input

### 시나리오 5-1. 사용자 미확인 상태 저장 차단
- 입력: 필수값 입력 후 `UserIdInput` 미확인
- 기대 결과:
  - DB 저장 버튼 비활성화
  - JSON 생성은 가능

### 시나리오 5-2. 중복 인물 감지
- 입력: 같은 사용자 계정에 동일 이름 + 생년월일 인물이 이미 존재
- 기대 결과:
  - 저장 전에 중복 확인 모달 표시
  - 기존 데이터 요약과 메모 확인 가능

### 시나리오 5-3. 메모 병합 후 저장
- 입력:
  - 기존 레코드 notes 존재
  - 새 입력에도 메모 존재
- 기대 결과:
  - 1건 중복일 때 "메모 병합 후 저장" 버튼 표시
  - 병합된 메모로 저장 가능

### 시나리오 5-4. 저장 후 split 검증
- 입력: ManualInputPage에서 DB 저장
- 기대 결과:
  - 저장 후 `fetchPersonById()` 재조회 수행
  - 일치면 success 메시지, 불일치면 경고 메시지 표시

## 6. 남은 Todo 검증 포인트

### 음력 윤달 처리 개선
- `birthDate.isLeapMonth`를 실제로 채울 수 있는 원본 신호가 있는지 확인
- 추론 불가 시 warning 또는 안내 문구가 필요한지 확인

### 병합 그룹 설명/추천 개선
- 사용자가 왜 병합되었는지 이해할 수 있는 설명이 현재 충분한지 확인
- 대표 레코드 선정 근거를 UI로 노출할 필요가 있는지 확인

### 병합 시 메모/노트 merge 품질 개선
- 동일 문구 중복 제거가 필요한지 확인
- `' / '` 대신 줄바꿈 또는 출처 표시가 더 적절한지 확인

### 저장 결과 CSV 내보내기
- 성공/실패/mismatch를 한 번에 정리하는 포맷이 필요한지 확인
- 엑셀에서 한글 인코딩이 깨지지 않는 형식을 선택해야 하는지 확인
